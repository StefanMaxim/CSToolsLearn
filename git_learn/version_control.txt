What does rebase actually do:
Fundamentally, given some stack of commits
A <- B <- C (HEAD), you are taking some commits from the stack, and replaying them on top of another commit 
(exe take B and C, and replay them onto some other stack)
D <- E <- B <- C (but its more precise than just that)
(same changes, different parents)

EXE:
main:
A <- B <- C 

feature:

A <- B <- D <- E (HEAD)

when you type: git checkout feature_branch, git rebase main:
first, it finds the fork point, B, 
then, take all commits on feature_branch after C (D and E)
then, it temporarily removes them
then, it replays them on top of main's head (B or newer, in this case C) as new commits D` and E`

result:
(recall, git merge branch1 will merge FROM branch1, meaning it will combine your changes from branch1 into your current_branch, likely main )
GIT MERGE BRANCH1, BRANCH1 DOES NOT CHANGE

(rebase is the same, it will rebase from the chosen one, so git rebase main doesnt change main!!)

main:

A <- B <- C (unchanged)

A <- B <- C <- D` <- E` (where D` is the new commit from C to B) (git rebase main = rebase head so that it is as if it came after main)
